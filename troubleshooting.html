Troubleshooting

문제를 재현하고 근본 원인을 해결한 과정

트러블슈팅은 단순히 오류를 수정하는 것이 아니라, 문제를 재현하고 원인을 분석한 뒤 개선 효과를 검증하는 과정이라고 생각합니다. 아래 사례들은 로그 분석, 실행 계획(EXPLAIN ANALYZE), 성능 테스트(JMeter), Mock 환경 등을 활용하여 원인을 확인하고, 개선 후 결과까지 검증한 경험을 정리했습니다.

파이널프로젝트

키워드 검색 성능 병목 개선

PostgreSQL
Problem
JMeter 성능 테스트 결과, 일반 채용공고 조회는 평균 9ms였지만 키워드 검색(keyword=Java)은 평균 342ms, p95 538ms로 응답 시간이 크게 증가했습니다. 키워드 검색 시 전체 서비스 응답 속도가 저하되어 실제 사용자 경험에 영향을 줄 수 있는 성능 병목이 발생했습니다.
Cause
EXPLAIN ANALYZE를 통해 실행 계획을 분석한 결과, 일반 목록 조회는 created_at 인덱스를 사용해 빠르게 처리되었지만, 키워드 검색은 여러 컬럼(title, description, company_name, source)과 배열(skill_tags, job_category)을 OR 조건으로 검색하고 array_to_string()을 사용하고 있었습니다. 이 구조 때문에 PostgreSQL이 일반 인덱스와 pg_trgm 인덱스를 활용하지 못하고 Parallel Seq Scan으로 전체 테이블을 반복 스캔했으며, COUNT(*)와 깊은 페이지 조회에서는 정렬과 디스크 Merge까지 발생하여 병목이 발생했습니다.
Solution
검색 대상 컬럼을 하나의 search_text 컬럼으로 통합하고, company_name, title, description, source, skill_tags, job_category 값을 저장하도록 변경했습니다. lower(search_text)에 GIN + pg_trgm 인덱스를 적용하여 기존의 복잡한 OR 조건을 단일 검색 조건으로 변경했고, Spring Boot와 FastAPI 저장 로직 모두에서 search_text가 자동 생성되도록 수정하여 검색 데이터의 일관성을 유지했습니다.
Result
실행 계획이 Parallel Seq Scan → Bitmap Index Scan으로 변경되었으며, 키워드 검색 성능이 크게 향상되었습니다. 깊은 랜덤 페이지 기준 JMeter 테스트 결과 Average: 342ms → 155ms (약 55% 개선) p95: 538ms → 239ms (약 56% 개선) Throughput: 2.9/sec → 6.4/sec (약 2.2배 향상) 실제 서비스와 동일한 무한 스크롤(초기 페이지) 환경에서는 평균 1ms, p95 2ms 수준으로 응답하여 사용자 체감 성능을 크게 개선했습니다. 또한 향후 매우 깊은 페이지 조회가 필요한 경우에는 Cursor Pagination으로 추가 개선할 수 있는 구조까지 고려했습니다.

파이널프로젝트

스크래핑 상세 데이터 완전성 및 성능 개선

FastAPI · Scraping
Problem
외부 채용 사이트의 상세 조회가 지연되거나 타임아웃될 경우, 상세 API와 HTML fallback을 순차적으로 모두 대기하면서 공고마다 응답 시간이 누적되었습니다. 또한 상세 데이터를 하나도 수집하지 못해도 공고 목록만 조회되면 파이프라인이 SUCCESS로 처리되어 데이터 품질을 보장하지 못했습니다.
Cause
기존 로직은 공고마다 상세 API → HTML fallback을 각각 독립적인 timeout으로 처리하여 동일한 요청에 대한 대기 시간이 반복적으로 발생했습니다. 또한 상세 조회 성공 여부를 검증하는 기준이 없어, 상세 데이터가 모두 누락된 경우에도 정상 수집으로 판단하는 구조였습니다.
Solution
상세 API와 HTML fallback이 하나의 공유 시간 예산(Time Budget) 을 사용하도록 변경하여 불필요한 대기 시간을 제거했습니다. Timeout, ConnectError, 429, 5xx 오류는 최대 1회만 재시도하도록 개선하고, 상세 조회의 시도·성공·실패·Timeout·재시도 지표를 수집하도록 구현했습니다. 또한 상세 조회 대상이 존재하지만 성공 건수가 0건인 경우 저장을 중단하고 파이프라인 상태를 SCRAPING_DETAIL_COMPLETENESS_FAILED로 기록하여 데이터 완전성을 검증하도록 개선했습니다.
Result
Mockoon 기반 테스트에서 상세 조회가 모두 실패하는 환경에서도 불필요한 대기 시간이 제거되어 성능이 향상되었습니다. 스크래퍼 실행 시간: 511.5ms → 257.0ms (약 50% 감소) 파이프라인 실행 시간: 464ms → 232.3ms (약 50% 감소) 또한 상세 데이터를 모두 수집하지 못한 경우 기존에는 SUCCESS로 처리되던 문제를 FAILED로 변경하여 잘못된 데이터 저장을 방지했습니다. 정상 환경에서는 상세 설명을 3/3건 모두 수집했으며, 약 92.7ms 내에 완료되는 것을 확인했습니다. 마지막으로 Timeout, ConnectError, HTTP 429/500 재시도와 상세 조회 실패 처리까지 포함한 테스트를 수행하여 48개의 테스트가 모두 통과했습니다.
